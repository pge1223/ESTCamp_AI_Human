## 근본 원인

### 1. 콜센터 정상 언어 ≈ 기관사칭 피싱 언어

```
피싱:  "본인 확인을 위해 성함과 생년월일을 말씀해 주세요"
정상:  "고객님 정보 확인 후 도움드리겠습니다. 성함 부탁드립니다"
```

K쇼핑 콜센터 언어("성함 확인", "정보 확인", "연락처 말씀")가
기관사칭 피싱 언어와 표면 패턴이 거의 동일 → 모델이 경계를 학습하지 못함.

v3 FP 상위 예시:
```
"네, 연락처 확인 부탁 드리겠습니다."       → 기관사칭 71.5% ❌
"전화주신 고객님 성함과 연락처를 말씀..."   → 기관사칭 74.7% ❌
"소중한 정보 확인 감사합니다."             → 기관사칭 83.7% ❌
```

### 2. 본질적인 문제 — 학습 단위 vs 실제 탐지 단위 불일치

현재 학습 데이터는 **단일 발화(문장/문단)** 단위이지만,
실제 피싱은 **대화 흐름 전체**에서 패턴이 드러납니다.

```
turn 1: "안녕하세요, 중앙지검 수사과입니다"       → 단독으로는 모호
turn 2: "고객님 명의 계좌에 이상거래 감지됐습니다" → 단독으로는 일반 안내처럼 보임
turn 3: "신분 확인을 위해 성함 부탁드립니다"      → 단독으로는 콜센터와 동일
turn 4: "안전계좌로 이체 필요합니다"              → 여기서 비로소 명확
```

turn 3만 보면 정상 콜센터와 구별 불가.
**turn 1~4 누적 흐름을 함께 봐야 피싱을 확신할 수 있음.**

그런데 `train_ml.json`은 각 turn을 독립 샘플로 잘라서 학습 →
모델이 "대화 흐름의 피싱 패턴"이 아닌 "문장 하나의 키워드 패턴"을 학습.

## 결론

| 항목 | v2 | v3 |
|---|---|---|
| 실전 성능 | ✅ 우수 (F1 93~96%) | ❌ 퇴보 (F1 19~29%) |
| 오탐률 | ✅ 낮음 (FP 7건) | ❌ 매우 높음 (FP 424건) |
| 미탐률 | ✅ 낮음 (FN 1건) | △ 수용 가능 (FN 7~13건) |
| 현재 배포 적합성 | ✅ | ❌ |

**v3는 정상 데이터를 더 많이 학습했지만, 잘못된 정상 데이터(콜센터)를 학습해서 오히려 퇴보.**
현재 기준 v2가 유일한 배포 후보.

## 대응 방향

### 단기 (즉시)

1. **config.py를 v2로 롤백** — 배포/테스트 기준 v2 모델 사용
2. **기관사칭 임계값 상향** — 0.5 → 0.7로 올려 오탐 추가 감소
3. **키워드 필터 보완** — `수사과`, `사무관`, `www`, URL패턴 등 추가 (완료)

### 중기 (데이터 재설계)

1. **학습 단위를 전체 통화로 변경**
   - 금감원 MP4 104개 → Whisper STT → **전체 통화 텍스트를 하나의 샘플**로 묶기
   - 지금처럼 문단 단위로 자르면 대화 흐름 소실

   ```json
   // 현재 (잘못된 방식)
   { "text": "성함 부탁드립니다", "label": [1, 0, 0] }

   // 목표 (대화 전체)
   { "text": "중앙지검입니다... 이상거래 감지... 성함 부탁... 안전계좌 이체", "label": [1, 1, 0] }
   ```

2. **normal_callcenter.json 재검토**
   - 피싱과 언어 패턴이 겹치는 샘플 필터링 후 재투입
   - 또는 콜센터 정상 데이터 대신 일상대화/뉴스 등 도메인이 명확히 다른 데이터로 교체

3. **평가 데이터 재구성 (v4 기준)**
   - 전체 데이터 합산 후 80/10/10 stratified split 적용
   - val/test에도 다양한 정상 패턴 포함시켜 evaluate.py가 실제 성능을 반영하도록
